Apples : Trees as JSObjects : Prototypes


A snarky explanatory example of JavaScript prototypes

11.21.2015

What is a JavaScript Prototype?

Prototype is a property of an object in Javascript (JS), and they are important when considering the concept of inheritance. All objects have a prototype property, which is empty by default. Inheritance is the process of making properties and methods available to instances of a function. It is accomplished by attaching properties and methods on to the prototype property. Here is an example of how inheritance works, using the prototype property.* 


        
  function SnarkyMessage (LovelyMessage) {
      this.LovelyMessage = LovelyMessage;
  }

  SnarkyMessage.prototype.snark = function () {
      console.log(this.LovelyMessage + " sucks");
  };

  var javaScriptRules = new SnarkyMessage ("I love JavaScript");

  javaScriptRules.snark (); 

  // output: "I love JavaScript sucks"

The function SnarkyMessage takes a property, LovelyMessage, which it defines as an instance of a variable called LovelyMessage. Just to be clear, this.LovelyMessage is an instance of the variable LovelyMessage. The function SnarkyMessage is then ascribed the snark method, using the prototype property. Note that if the prototype property is removed, running the program raises the following error message: TypeError: javaScriptRules.snark is not a function. This indicates that using the prototype property is necessary in order to call .snark on this.LovelyMessage. Why? because this.LovelyMessage is an instance of SnarkyMessage (LovelyMessage). A new object can now access the results of the SnarkyMessage.prototype.snark function, in the following manner: declare a new object to be a new instance of SnarkyMessage (including the variable, which will be the parameter operated upon). Call .snark on the new object.


In short, the prototype attribute is telling us where javaScriptRules got its “snark” from: it’s parent, the SnarkyMessage function. This demonstrates an odd similarity between humans and JS objects: we all eventually turn into our parents. At least JS objects are up front about it, and the characteristic doesn’t suddenly dawn on them when they’re in their 30s.


This example uses object literal notation to show how inheritance works via prototype, but prototype can also be used with a constructor function to create an object. What else is there to know about prototypes? How about the prototype chain - this tells us where an object’s inherited properties were originally defined. If the property is not defined in the object, JS will look at its parent object. If it can’t find the property there, it looks at its “grandparent” object, going all the way back up the prototype chain. If JS cannot find where the property is defined anywhere in the chain, can you guess what gets returned? Yep, undefined.


In sum, the big takeaway regarding prototype is understanding how inheritance works on JS objects. By using prototype, you can define methods, via “parent methods,” on instances of an object, as shown in the snarky example above.
